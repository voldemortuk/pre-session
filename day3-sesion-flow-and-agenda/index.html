What Today Is

A build day, not a lecture.

Everything you touch today runs on a live Command Center build — Units, Alerts, Clips, deterrent controls, multi-tenant Orgs — modelled on LVT’s own system. This isn’t a toy app standing in for your job. It’s your domain, and by the end of the day you’ll have driven real review, security and permission gates over it with your own hands.

An AI first draft that looks idiomatic is the most dangerous kind of code — because tidiness disarms the reviewer.

So today we stop trusting how code reads and start trusting what it passes. Four gates, built by you: a self-verifying loop, independent review, secure-by-default tooling, and a committed permissions standard. Review effort is a budget — spend it on what only a human can catch, and move everything a machine can catch onto a deterministic gate.

What You’ll Do — Part by Part

The four builds, in detail.

Each part is a short concept, then a lab you drive yourself. Here’s what you’ll learn and, more importantly, what you’ll actually build with your own hands.

01

Anti-Slop & Self-Verify

Make review of AI code calibrated, not exhausting

  • What “slop” really is: plausible-but-wrong code — inverted comparisons, copy-paste drift, dead branches, tests that assert nothing, off-by-one. It reads clean, so a normal read-through misses it.
  • Why veterans miss it, and the fix: a scored 5-pass checklist plus a self-verifying loop that runs build + test + lint before you ever see the diff.
🖥 Hands-on — you will 35 min
  • Run make slop to plant 8 real defects into an “acknowledge alerts” feature on the Command Center build.
  • Review the diff solo against the checklist, scoring a point per defect you spot and explain — then pair up and compare scores.
  • Run /self-verify and see which defects the deterministic layer catches for free, and which still need your judgment.
02

Adversarial Review: Multi-Hat Thinking

Get judgments that aren’t the author’s

  • Why one reviewer wearing every hat — security, design, performance — does each one shallowly.
  • How independent reviewers in separate, non-cross-talking sessions stay sharp and are allowed to disagree — and why the black-box hat, which sees only the API contract, catches bugs the tidy-code reviewer never does.
🖥 Hands-on — you will 40 min
  • Run a Security hat and a Black-box hat in their own sessions, capture each verbatim, then line them up against a single all-in-one reviewer.
  • Watch the black-box hat catch a cross-org data leak from the contract alone — the one the blended reviewer smooths right over.
  • Write one human-authored test on the critical path: it fails today, passes after Part 3’s fix. The rule to keep: never ship AI code with only AI-written tests.
03

Security: Secure-by-Default & OWASP

Catch issues while the agent writes — and keep the tooling honest

  • The five vulnerability classes in today’s build: SQL injection, IDOR, SSRF, stored XSS, secrets/misconfig — and the one idea under all of them: authenticated ≠ authorized.
  • Why a tool’s flag is a claim, not a verdict, and the risk framing for the skills you install: a skill runs with your agent’s privileges — the “lethal trifecta” of reading private data, seeing untrusted input, and reaching the network.
🖥 Hands-on — four activities 60 min
  • Tool pass: run the owasp-review skill over the API and the alert-notes rendering, and record what it flags.
  • Independent pass: run the security-reviewer subagent on multi-tenant scoping — it names the missing cross-tenant test.
  • Triage one finding end to end: prove a real cross-org leak with two org tokens, and write down the verdict — a fix and a dismissal both need a written record.
  • Everyone runs the hook: add a planted AWS key to the config and watch your own secret-scan hook deny it — the most concrete deterministic gate of the day.
04

Permissions Without Constant Interruptions

Flow without footguns — committed as a team standard

  • The three lists in .claude/settings.json — allow / ask / deny, checked in that order, first match wins, a broad deny beats a narrow allow.
  • That rules are enforced by Claude Code, not the AI — “don’t do X” in a prompt is a request, not a fence — plus the scopes (user / project / local / managed) and defense in depth.
🖥 Hands-on — you will 45 min
  • Inspect every active rule with /permissions and validate the file with claude doctor.
  • Break three rules that look safe — find a command each fails to stop, then rewrite it so it holds.
  • Layer a gate: protect .env with a deny rule, watch a script slip past it, then add a hook that catches what the rule can’t.
  • Decide & commit: debate a contested rule, record the decision and the dissent, commit the team’s settings.json, and watch the real CI pipeline run on your own PR.
